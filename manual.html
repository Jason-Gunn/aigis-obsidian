AIGIS Governance

A stripped-down AI governance layer for Obsidian. Register models, manage prompts and policies, document workflows, store governed skills, track incidents, and maintain an append-only audit log — all inside a Markdown vault.

What is AIGIS?

AIGIS Governance is an Obsidian community plugin that brings structured AI governance workflows into your note-taking environment. It is a focused adaptation of the AIGIS WordPress plugin, keeping the modules that map naturally to Markdown notes while omitting the parts that require a live server (live API sandbox, usage analytics, cost budgets, REST API).

The result is a lightweight but opinionated governance workspace: every AI model you operate, every governed prompt, every formal policy, every approved workflow, every reusable skill, and every incident you investigate lives as a structured Markdown note with consistent frontmatter. An append-only audit note records every action the plugin takes.

What it is for:

  • Individuals and small teams who want rigour around their AI tooling without running a full platform.
  • Compliance-track organisations who need a documented evidence base and prefer to own their data locally.
  • Prompt engineers and AI developers who want version-controlled, lifecycle-managed prompt artefacts.

What it is not:

  • It does not call any AI provider or run prompts — it is purely a governance record.
  • It does not provide real-time usage analytics or cost tracking (those require a server-side component).
  • It is not a replacement for the full AIGIS WordPress plugin if you need a multi-user web interface.

Vault Structure

The plugin manages a single root folder inside your vault (default: AIGIS/). Run AIGIS: Bootstrap governance vault once after enabling the plugin to create the full structure and the dashboard.

AIGIS/ ├── Dashboard.md # auto-generated, regenerated on every change ├── Inventory/ │ └── your-model-record.md ├── Prompts/ │ └── policy-summariser.md ├── Policies/ │ └── acceptable-use-policy.md ├── Workflows/ │ └── model-deployment-approval.md ├── Skills/ │ └── complaint-triage.md ├── Incidents/ │ └── prompt-injection-blocked.md └── Audit/ └── Audit Log.md # append-only, never edited by hand
💡
The root folder name, audit folder name, and dashboard filename are all configurable in Settings → AIGIS Governance. You can change them before or after bootstrapping — just update the setting and re-bootstrap to recreate the structure in the new location.

AI Inventory

The Inventory module is the foundation of the governance workspace. Every other module can reference an inventory record by linking to its note. Before creating prompts, policies, or skills, register the AI systems they relate to here.

Fields

FieldDescription
vendorThe organisation or provider operating the model (e.g. OpenAI, Anthropic, Meta).
modelModel name and version (e.g. gpt-4.1, claude-opus-4, llama3:8b).
integration_typeapi-model — cloud API access. custom-agent — bespoke orchestration. on-prem — self-hosted (e.g. via Ollama).
risk_levellow, medium, or high based on data sensitivity and blast radius if misused.
ownerTeam or person responsible for this model deployment.
statusactive, deprecated, or under-review. Deprecated records are kept for audit linkage.

Note sections

The generated note includes Overview (scope and purpose), Governance Notes (data categories, human review expectations, linked policies), and Operational Context (agent identifier, endpoint, dependencies).

Prompts

Prompts are system instructions, templates, or personas used with AI models. AIGIS treats them as version-controlled artefacts with a defined lifecycle stage.

Lifecycle stages

  • Development — in active iteration, not yet approved for use.
  • Staging — review-ready, being tested before production clearance.
  • Production — approved and cleared for use in governed workflows.

Update the stage frontmatter field manually as a prompt progresses. Promotion is a deliberate, recorded act — the audit log captures the timestamp and the old and new stage values when you edit the field.

Note sections

Generated notes include Purpose (populated from the Goal field), Prompt Body (paste the actual prompt text here), and Review Notes (promotion criteria, edge cases, linked inventory record).

Policies

Policies are formal governance documents that define permitted uses of AI, prohibitions, and consequences of non-compliance. Obsidian's built-in Dataview plugin (if installed) can query the review_date frontmatter field to surface upcoming review obligations. The auto-generated dashboard does this natively for the top five upcoming reviews.

Status values

  • draft — under authorship, not yet reviewed.
  • approved — formally signed off and in effect.
  • retired — superseded or no longer applicable; retained for historical record.

Best practice

  • Always assign an owner field so responsibility is traceable.
  • Set a review_date no more than 12 months from the effective_date.
  • Cross-link policies to the inventory records they govern using Obsidian wiki-links.

Workflows

Workflow notes document approved AI-assisted processes: the sequence of steps, where human oversight is mandatory, and what automated decisions are permissible. Each note is generated with a Mermaid diagram block pre-populated from the form.

Mermaid diagrams

If you have the Mermaid community plugin installed (or use an Obsidian version that renders Mermaid natively), the diagram block renders inline as a flowchart. The form pre-populates a simple starter diagram which you can replace with any valid Mermaid syntax.

```mermaid
graph TD
    A[Draft Workflow] --> B[Governance Review]
    B --> C{Approved?}
    C -->|Yes| D[Release to Production]
    C -->|No| E[Rework]
```

Best practice

  • Mark every node where a human must review output before proceeding.
  • Note in the Oversight Notes section what evidence a reviewer needs before they can sign off.
  • Cross-link to the relevant Policy notes that authorise the workflow.

Skills

A skill is a reusable governed capability — an instruction bundle that tells an agent what to do, when to do it, and what its output must look like. Skills are the bridge between your governance policies and the actual instructions you give to AI systems.

Key fields

FieldPurpose
trigger_phrasesThe phrases or conditions that should cause an agent to invoke this skill. One phrase per line.
output_contractThe mandatory structure of the skill's output. State exact section names or formats required.
teamThe team that owns and is responsible for reviewing this skill.
versionA version tag so you can track iterations (e.g. 1.0.0).

Note sections

Generated notes include Capability Summary, Method (the actual instruction body), and Readiness Review (edge cases, linked prompts, workflows, policies, and incidents).

ℹ️
Always populate the output contract before using a skill in production. Vague output contracts are the most common cause of inconsistent agent behaviour.

Incidents

Log and investigate AI incidents here — PII leakage in outputs, prompt injection attempts, misuse, or unexpected model behaviour. The structured format ensures nothing falls through the cracks.

Severity levels

  • low — unexpected but low-impact behaviour with no data or user harm.
  • medium — potential compliance exposure or repeated boundary violations.
  • high — confirmed data exposure, harmful content, or regulatory risk.
  • critical — active harm, confirmed PII breach, or immediate escalation required.

Status flow

open → investigating → resolved. Update the status frontmatter field manually as the investigation progresses. Link the incident to inventory records and skill notes for full traceability.

Note sections

Generated notes include Incident Summary (populated from the form), Investigation (detection time, containment, root cause), and Resolution (owner, follow-up actions, control changes required).

Audit Log

The audit log at AIGIS/Audit/Audit Log.md is an append-only record written by the plugin. You should never manually delete lines from it. The plugin appends a line for every consequential action it takes.

Format

- 2026-04-22T14:33:01.000Z | vault.bootstrap | AIGIS governance vault bootstrapped.
- 2026-04-22T14:35:20.000Z | note.created | Inventory note created: Customer Support GPT | module=inventory, file=AIGIS/Inventory/customer-support-gpt.md

Each line contains: ISO-8601 timestamp, action type, human-readable summary, and optional key=value metadata pairs.

Logged actions

  • vault.bootstrap — vault structure created or re-bootstrapped.
  • note.created — any governance note created through the plugin (includes module, file path).
⚠️
The audit log is a plain Markdown file. Obsidian does not prevent editing it. Treat it as advisory evidence rather than a cryptographic audit trail. For regulatory use cases, export the file regularly and store a copy outside the vault.

Dashboard

The AIGIS/Dashboard.md note is regenerated automatically by the plugin every time a managed note is created, deleted, or renamed. It contains:

  • A count of notes in each module folder with a wiki-link to the folder.
  • Quick links to the audit log and workspace README.
  • The five nearest upcoming policy review_date values, sorted chronologically and linked to the relevant policy note.

Open the dashboard note and pin it in Obsidian as a persistent workspace overview. Use it as the starting point for any governance review session.

Console Panel

The AIGIS Console is a persistent side panel in Obsidian's right sidebar. It opens automatically on startup (configurable) and provides:

  • Three quick-action buttons: Bootstrap vault, Open dashboard, Open audit log.
  • One card per module showing the live note count and a Create button that opens the creation modal.

The panel refreshes its counts whenever a note is created, deleted, or renamed inside the managed root folder. Open it any time via the shield icon in the ribbon or with the AIGIS: Open governance console command.

Installation

There are two paths depending on whether you are starting a fresh vault or adding the plugin to an existing one.

Option A — Open this repo as a new vault

If you have cloned or downloaded the AIGIS-Obsidian repository, the folder is already a self-contained vault with the plugin source inside it.

  1. Build the plugin In a terminal, navigate to the plugin folder and install dependencies then build: 
    cd ".obsidian/plugins/aigis-governance"
npm install
npm run build
    This produces main.js inside the plugin folder.
  2. Open the folder as a vault in Obsidian Launch Obsidian → Open another vault → Open folder as vault → select the AIGIS-Obsidian folder.
  3. Enable the plugin Go to Settings → Community Plugins. If prompted, turn off Restricted Mode. Find AIGIS Governance in the list and toggle it on.
  4. Bootstrap the vault Open the command palette (Ctrl/Cmd + P) and run AIGIS: Bootstrap governance vault. The plugin creates the folder structure, the dashboard note, and the audit log, then opens the console panel.
💡
You do not need to keep your governance notes inside the AIGIS-Obsidian folder. Once the plugin is built you can copy just the three plugin files (main.js, manifest.json, styles.css) into any vault.

Option B — Add to an existing vault

If you already use Obsidian with a main vault and want AIGIS to run inside it, copy the compiled plugin files into your vault's plugin directory.

  1. Build the plugin (if not already done) 
    cd ".obsidian/plugins/aigis-governance"
npm install
npm run build
  2. Locate your existing vault's plugin folder In your main vault, navigate to <your-vault>/.obsidian/plugins/. Create it if it does not exist.
  3. Copy the three plugin files Copy these three files from the repository into a new sub-folder named aigis-governance inside your vault's plugins directory: 
    .obsidian/plugins/aigis-governance/
├── main.js          ← compiled output
├── manifest.json    ← plugin metadata
└── styles.css       ← console and modal styles
  4. Enable the plugin In Obsidian, go to Settings → Community Plugins → Reload plugins, then find AIGIS Governance and enable it.
  5. Configure the root folder (optional) Go to Settings → AIGIS Governance and set the Root folder to wherever you want governance notes to live inside your existing vault (e.g. Work/AI Governance).
  6. Bootstrap Run AIGIS: Bootstrap governance vault from the command palette to create the folder structure inside your vault.
⚠️
If you already have a folder at the configured root path, bootstrapping will add missing sub-folders and files without overwriting existing ones. It will not delete or modify any notes that are already there.

Development mode

If you want to modify the plugin source, run the watcher instead of a one-shot build. The watcher rebuilds main.js on every file save with inline source maps for easier debugging.

cd ".obsidian/plugins/aigis-governance"
npm run dev

After saving a change, use Ctrl/Cmd + P → Reload app without saving in Obsidian to hot-reload the plugin, or toggle it off and on in Settings → Community Plugins.

ℹ️
The src/main.ts file contains the entire plugin. There is intentionally no multi-file split at MVP scope — all types, commands, views, modals, and settings live in one file for simplicity.

Command Reference

All commands are available from the command palette (Ctrl/Cmd + P). You can also assign hotkeys in Settings → Hotkeys.

CommandDescription
AIGIS: Bootstrap governance vaultCreates the folder structure, audit log, and dashboard if they do not exist. Safe to re-run — nothing is overwritten.
AIGIS: Open governance consoleOpens or focuses the AIGIS Console side panel in the right sidebar.
AIGIS: Create Inventory record noteOpens the inventory creation form and places the new note in AIGIS/Inventory/.
AIGIS: Create Prompt noteOpens the prompt creation form and places the note in AIGIS/Prompts/.
AIGIS: Create Policy noteOpens the policy creation form and places the note in AIGIS/Policies/.
AIGIS: Create Workflow noteOpens the workflow creation form (includes a Mermaid starter diagram) and places the note in AIGIS/Workflows/.
AIGIS: Create Skill noteOpens the skill creation form and places the note in AIGIS/Skills/.
AIGIS: Create Incident noteOpens the incident creation form and places the note in AIGIS/Incidents/.

Frontmatter Reference

Every note created by the plugin includes the following system-level frontmatter fields in addition to the module-specific ones:

FieldPresent inDescription
aigis_moduleAll notesThe module type: inventory, prompts, policies, workflows, skills, or incidents. Useful for Dataview queries.
titleAll notesThe human-readable title entered in the creation form.
created_atAll notesISO-8601 timestamp written at creation time.
updated_atAll notesISO-8601 timestamp at creation (not auto-updated on edit — update manually or via a Templater script).

All other frontmatter fields are module-specific and match the form fields described in each module section above.

Example: querying with Dataview

If you have the Dataview plugin installed, you can write queries like:

```dataview
TABLE vendor, model, risk_level, status
FROM "AIGIS/Inventory"
SORT risk_level DESC
```
```dataview
TABLE status, review_date, owner
FROM "AIGIS/Policies"
WHERE review_date <= date(today) + dur(60 days)
SORT review_date ASC
```
